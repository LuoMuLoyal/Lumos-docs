# Luminous 项目愿景文档

Last updated: 2026-06-09

## 一句话定位

Luminous 是面向大学生的日常健康自我管理助手，以用药安全为可信入口，连接症状、饮水、饮食、睡眠和用药记录，在用户授权和数据足够时通过 AI 总结、趋势分析和主动提醒，帮助用户在校园高频健康场景中做出更安全、更有依据的下一步行动。

## 这次定位调整的结论

Luminous 不应该被限制成单纯的“药品查询工具”或“用药安全助手”。用药安全仍然是最硬的切入点，因为它高频、具体、风险真实，也最容易形成可信闭环。但产品的长期价值不只在“查药”，而在于把用户每天分散的健康记录连接起来，主动发现值得注意的变化、生成总结、给出低风险生活建议，并在需要时把用户引导到校医院、药师或其他校园支持资源。

新的方向不是“什么健康场景都做”，而是：

```text
记录 -> 结构化 -> 总结/提醒 -> 安全解释 -> 服务连接/报告
```

核心变化：

- 保留用药安全作为可信入口。
- 增加日常记录分析、每日总结、每周趋势、主动提醒，但所有 AI 输出必须受数据授权、来源和安全边界约束。
- 手机端是主产品，Web 端只做报告预览、导出、演示和轻量查看；真实导出和分享必须等报告与隐私合同稳定。
- 砍掉运动恢复、女性健康、硬件、桌面端、多端扩展。
- 心理健康作为未来扩展，不放进 MVP 主线。
- AI 做记录理解、总结、解释和提醒文案，不做诊断、处方、药物风险判定或自动调整用药方案。
- 药物风险判断必须来自规则库、药品说明书、可信数据源或人工审校数据；跨来源药物匹配、重复成分识别和未审校相互作用不进入未验证 MVP。

## 目标用户

第一阶段用户是中国高校里的大学生，尤其是：

- 经常感冒、发烧、头痛、过敏或胃肠不适，需要自己买药或用药的学生。
- 作息不稳定、饮水少、饮食不规律、经常熬夜的学生。
- 不愿意每次小问题都去医院，但又担心乱用药、漏服、重复用药或药物冲突的学生。
- 希望把近期健康情况整理成简洁摘要，带去校医院、药房或咨询医生时使用的学生。

第一阶段不把老人、儿童、慢病患者、家庭协作、医疗机构管理或专业诊疗作为主要用户。

## 核心问题

常规健康类产品往往是被动的：

- 用户身体不舒服了，才去搜索网页。
- 用户想知道药物信息了，才去查药。
- 用户记录数据后，很少有人帮他理解这些记录。
- 用户忘记吃药、饮水不足、熬夜变多时，系统通常只会机械提醒，不会结合上下文解释。

Luminous 要解决的问题不是“再做一个 AI 问答入口”，而是：

> 能否基于大学生每天真实产生的健康记录，主动发现值得注意的变化和风险，并给出可解释、低风险、可执行的下一步建议。

## 产品主张

Luminous 的产品主张是“主动式校园健康助手”。

它不是等待用户来问的搜索框，而是围绕用户已经记录或授权的数据，持续完成三件事：

1. **理解当天发生了什么。**  
   例如用户记录了头痛、早餐很晚、午饭油腻、饮水少、晚上喝咖啡、服用布洛芬等信息，系统需要把这些记录结构化，并识别出值得提醒的点。

2. **用人能理解的方式总结和解释。**  
   例如“今日饮水记录偏少，晚间记录了咖啡因，可能影响睡眠”，而不是只展示一堆数字或从粗粒度记录推断精确营养结论。

3. **在合适时间主动提醒。**  
   例如到服药时间仍未确认时提醒用户核对是否已服药；新增药物后，先检查可用的说明书、过敏史和已记录用药，信息不足时明确提示“不确定”。

## 平台策略

| 平台 | 定位 | MVP 状态 |
| --- | --- | --- |
| 手机端 | 主产品。负责记录、提醒、用药安全、今日建议、隐私控制和校园服务入口。 | 保留 |
| Web 端 | 报告查看、导出预览、打印、竞赛演示和轻量分享。真实导出/分享受报告与隐私合同门控。 | 有限保留 |
| 桌面端 | 不符合大学生日常碎片化记录和即时提醒习惯。 | 砍掉 |
| 手表、音箱、硬件 | 增加复杂度，不能证明核心假设。 | 砍掉 |
| 多端复杂同步 | 对 MVP 不是核心证明点。 | 砍掉 |

## MVP 范围

MVP 要证明一件事：

> 大学生愿意持续记录基础健康信息，并且认为 Luminous 的主动总结、提醒和用药风险解释比自己临时搜索更有用。

MVP 包含的产品闭环：

- 症状记录。
- 饮水记录。
- 饮食记录，先支持粗粒度描述，不追求精确营养计算。
- 睡眠记录；MVP 允许结构化记录、回看和趋势汇总，但不扩展成更重的睡眠健康专项模块。
- 用药记录。
- 个人用药盒。
- 用药提醒，包括时间计划、确认已服/漏服和记录漏服事件。
- 新增药物后的安全检查提醒，但只覆盖已有来源可支持的说明书、过敏史、当前用药和已审校规则。
- 每日 AI 总结，前提是用户授权且当天有足够记录。
- 每周趋势分析，前提是有足够连续记录和稳定报告数据合同；没有合同前只能作为静态演示或占位。
- 今日主动建议流，建议必须绑定用户记录、规则或可信来源。
- Web 报告预览和导出演示；真实导出、打印和分享必须等报告、隐私和后端合同稳定。

合同门控项不应作为已完成 MVP 能力宣传：

- 真实报告洞察、趋势图、导出文件和分享链接。
- 后端同步的隐私授权、通知偏好、提醒偏好和校园服务配置。
- 用户可直接触发的真实报告导出/分享动作。
- 药物跨来源匹配、重复成分自动识别、未审校药物相互作用判断。
- 红旗信号规则库、服务资源来源、固定安全文案和地图/校园服务发现。
- 语音、截图等更重输入形态下的自然语言到结构化记录写入链路。

MVP 不包含：

- 运动恢复。
- 女性健康、经期管理。
- 心理健康正式模块。
- 智能硬件。
- 桌面端。
- 家庭协作。
- 儿童用药。
- 慢病完整管理。
- 皮肤识别、影像识别或其他诊断能力。
- 自动处方、自动调整药量、自动改变服药方案。
- 没有规则、说明书、可信来源或人工审校支撑的药物安全判断。
- AI 自行判断红旗信号、疾病风险分级或就医紧急程度。
- 基于 agent 或地图服务的动态附近就医/药房发现。

心理健康可以作为未来扩展，但第一阶段只允许保留非常轻量的非诊断状态标签或校园支持资源入口，不把它作为核心场景，也不命名为正式心理健康模块。

## AI 能力设计

AI 应该被放在“理解、总结、解释、提醒文案”层，而不是“诊断、处方、替代医生或药物风险判定”层。

| 能力 | 说明 | MVP 建议 |
| --- | --- | --- |
| 自然语言转结构化记录 | 用户输入“今天头疼，早上喝了两杯水，晚上没睡好”，AI 拆成候选记录。用户确认前不写入正式记录。第一阶段只做文本输入到底部弹层候选确认，不在 MVP 内承诺语音、截图或药物自动识别。 | 第一阶段做文本候选确认版 |
| 每日总结 | 汇总当天饮水、饮食、睡眠、症状、用药，指出最值得注意的 1 到 3 个点。只使用用户授权和已记录数据。 | 做 |
| 每周趋势分析 | 对比本周和上周，例如饮水增加、睡眠减少 11.2%、晚间咖啡因增多。只有记录密度足够且报告合同稳定时才生成真实趋势。 | 门控做 |
| 主动提醒文案生成 | 根据记录、提醒计划或规则生成更自然的提醒文案，而不是机械通知。 | 做 |
| 用药风险解释 | 风险判断来自规则库、药品说明书、可信药物数据或人工审校数据，AI 只负责解释原因和注意事项。 | 做，但必须有规则兜底 |
| 校医院摘要生成 | 生成给校医院、药师或医生看的近期摘要，用户可预览并选择隐藏敏感内容。真实分享/导出受隐私和报告合同门控。 | 门控做 |
| 红旗信号提示 | 只基于固定规则表和固定安全文案，对高烧不退、严重过敏、呼吸困难等记录提示尽快寻求线下帮助。AI 不自行判定紧急程度。 | 规则库完成后做 |
| 个性化目标建议 | 例如建议明天多喝水、早点睡、减少晚间咖啡因。建议必须绑定可观察记录，不从粗粒度饮食描述推断精确营养结论。 | 做 |
| 疾病判断 | 判断用户得了什么病。 | 不做 |
| 药量调整 | 建议加量、减量、停药。 | 不做 |
| 漏服后自动改服药时间 | 直接计算“下一次几点吃、晚上延后到几点”。 | 不做 |
| 自动处方 | 给出具体处方或替代医生开药。 | 不做 |

## 每日总结

每日总结是 Luminous 从“记录工具”变成“健康助手”的关键能力。

示例输出：

```text
今日饮水记录偏少，晚间记录了咖啡因，可能影响睡眠。
明日建议适当增加饮水，晚间尽量减少咖啡因摄入。
如果头痛或胃部不适持续，请及时咨询医生或校医院。
```

每日总结原则：

- 只总结用户已经记录或授权的数据。
- 优先指出 1 到 3 个最重要的变化，不写长篇泛泛建议。
- 建议要低风险、可执行，例如饮水、休息、清淡饮食、减少晚间咖啡因。
- 不从粗粒度饮食描述推断精确热量、蛋白质、碳水或脂肪结论，除非用户明确记录了这些数据。
- 涉及症状持续、严重不适或药物风险时，引导咨询医生、药师或校医院。
- 不写诊断结论。

## 每周趋势分析

每周分析用于帮助用户看到变化，而不是制造焦虑。

示例输出：

```text
本周饮水量比上周更多，继续保持。
本周平均睡眠时间比上周减少 11.2%，建议下周优先观察入睡时间和白天疲劳感。
本周有 3 天在晚上记录咖啡因摄入，可能与睡眠下降有关。
```

每周分析原则：

- 只比较有足够记录的数据。
- 明确“比上周增加/减少多少”，不要只说“变好了/变差了”。
- 把建议限定在生活方式和自我观察。
- 不把趋势直接解释成疾病。
- Web 端可以重点展示每周报告、趋势图、导出预览和打印预览；真实导出和分享必须等报告与隐私合同稳定。

## 主动建议与提醒系统

主动建议系统是 Luminous 相比普通健康 App 的核心差异。

普通健康产品大多依赖用户主动搜索。Luminous 应该在用户没有主动提问时，根据上下文发现问题并提醒。

可做的提醒类型：

- **饮水提醒**：连续饮水不足或当天饮水明显偏少时，提高提醒优先级。
- **饮食提醒**：基于用户明确记录的饮食时间、饱腹感或不适，提示清淡饮食、观察胃肠反应等低风险建议；不替用户决定是否延后进食。
- **睡眠提醒**：连续晚睡、睡眠下降、晚间咖啡因增加时，提醒观察睡眠。睡眠合同未稳定前只做占位或静态演示。
- **用药提醒**：到点未打卡时提醒用户确认是否已服药。
- **新药风险检查**：用户新增药物后，主动检查已有来源可支持的说明书、过敏史、当前用药和已审校规则；跨来源匹配、重复成分和未审校相互作用不作为确定结论。
- **报告提醒**：每周生成摘要后，提醒用户查看本周变化。没有真实报告合同时只做静态演示。
- **服务连接提醒**：后续可在固定规则、稳定资源合同或地图能力完成后，再提示校医院、药房、急诊或附近线下支持资源；这不属于当前 MVP 承诺。

需要避免的提醒：

- 不要替用户决定疾病诊断。
- 不要替用户决定是否停药、加药、换药。
- 不要在漏服后直接自动改用药方案。
- 不要用恐吓式语言制造健康焦虑。
- 不要把低置信度推断包装成确定事实。
- 不要在缺少饮食、睡眠、药物或症状数据时生成看似个性化的建议。

## 用药安全边界

用药安全仍然是 Luminous 的可信基础，但必须把 AI 放在正确位置。

推荐设计：

- 药物相互作用、重复成分、过敏禁忌、酒精/咖啡因冲突等风险判断，必须来自规则库、药品说明书、公开可信资料或人工审校数据。
- AI 负责把风险解释成用户能理解的话。
- 每条风险解释尽量展示来源、适用条件和不确定性。
- 如果信息不足，系统应该说“不确定”，并引导用户咨询医生或药师。
- Phase 1 不自动合并中文商品药和 DrugBank 药物实体。跨来源匹配、重复成分识别和相互作用扩展必须等成分归一化或人工审校数据稳定后再进入真实判断链路。

漏服场景的边界：

- 可以提醒用户“疑似漏服，请确认是否已经服药”。
- 可以记录漏服事件。
- 可以展示药品说明书中已有的漏服处理说明。
- 可以提醒用户不要自行加量或合并服用。
- 不应由 AI 自动决定新的服药时间、剂量或医嘱变化。

红旗信号的边界：

- 可以用固定规则表识别明确高风险表达，例如呼吸困难、严重过敏反应、高烧持续不退、意识异常等。
- 命中规则后只能提示“建议尽快寻求线下医疗帮助”，并提供校医院、急诊或药师资源入口。
- 不应由 AI 自行判断紧急程度、分诊等级、疾病名称或是否可以继续观察。
- 如果红旗规则、资源来源或固定安全文案没有完成审校，只能作为静态演示，不作为真实用户能力。
- 当前 MVP 不把这条能力作为完成标准。

## AI 与隐私边界

AI 总结、报告摘要和校医院/药师摘要都必须遵守最小必要原则。

- 用户未授权时，不把个人健康记录发送给 AI 或生成对外摘要。
- 摘要生成前应让用户预览，分享或导出前必须由用户确认。
- 过敏史、当前用药、症状、心理/情绪状态、备注和图片附件等敏感字段应支持隐藏或脱敏。
- AI 输出只能引用用户已记录、已授权或有来源支撑的信息；不能补写用户没有记录的事实。
- 报告分享、真实导出和跨设备隐私偏好同步必须等后端合同稳定后再作为真实能力。

## 信息架构建议

五个底部 tab 仍然合理，但每个 tab 的职责要收窄。

| Tab | 职责 |
| --- | --- |
| 今日 | 主动提醒、今日任务、AI 日总结、需要立即处理的规则命中风险和建议。 |
| 记录 | 快速记录症状、饮水、饮食、睡眠、用药，以及自然语言候选记录入口。 |
| 用药 | 药品查询、个人用药盒、用药提醒、来源支撑的安全解释。 |
| 报告 | 每日/每周/月度总结、趋势分析、Web 报告预览；真实导出受合同门控。 |
| 我的 | 健康档案、过敏史、隐私设置、通知设置、校园服务资源、账号设置。 |

关键边界：

- `今日` 不做深度录入，只承接主动提醒和轻量动作。
- `记录` 负责原始事实，不负责分析结论。
- `用药` 负责药物相关安全边界。
- `报告` 负责回顾性总结和趋势。
- `我的` 负责身份、隐私、服务资源和低频设置。

## 竞赛叙事

竞赛里不要把 Luminous 讲成“全能 AI 健康管理平台”。这个说法太大，也容易被质疑合规和落地。

更稳的说法：

> Luminous 是面向大学生的主动式健康助手。它以用药安全为可信入口，把日常症状、饮水、饮食、睡眠和用药记录连接起来，在用户授权、数据足够且规则来源明确时生成每日总结、每周趋势和主动提醒，帮助学生在校园常见健康场景中少一点乱搜、少一点漏服、少一点误用药，并在需要时连接校医院和药师资源。

差异化表达：

- 不是单点药品查询，而是“用药安全 + 日常记录 + 主动提醒”的闭环。
- 不是泛健康数据看板，而是围绕大学生高频场景做低风险建议。
- 不是 AI 医生，而是带安全边界的记录理解、风险解释和提醒助手。
- 不是用户主动搜索才响应，而是系统主动发现值得注意的变化。

## 第一阶段演示场景

### 场景一：新增药物后的主动风险提醒

用户把一种新药加入用药盒。Luminous 先检查已有来源可支持的药品说明书、过敏史、当前用药和已审校规则。如果命中明确风险，系统在 `今日` 和 `用药` 中提醒，并用可解释语言说明来源、适用条件和不确定性；如果信息不足，系统明确说“不确定”。

成功展示点：

- 用户不是主动搜索风险，而是系统在新药加入后主动检查。
- 风险解释不是黑箱 AI 猜测，而是有规则和来源支撑。
- 跨来源匹配、重复成分和未审校相互作用不包装成确定结论。
- 输出保持非诊断边界。

### 场景二：当天记录后的 AI 总结

用户记录了饮水少、晚间咖啡因、睡眠不足和轻度头痛。Luminous 在当天结束时生成简短总结，并把第二天最重要的低风险建议放到 `今日`。

成功展示点：

- AI 能把零散记录变成用户能理解的总结。
- 建议具体、低风险、可执行。
- 不从粗粒度饮食描述推断精确营养结论。
- 不夸大为医疗判断。

### 场景三：每周趋势报告

用户打开 `报告` 或 Web 报告页，看到本周与上周的变化，例如饮水增加、睡眠下降 11.2%、晚间咖啡因更多。Luminous 在记录密度足够且报告合同稳定时给出趋势解释和下周建议；否则只展示静态演示或占位状态。

成功展示点：

- Web 端承担报告预览和导出演示，而不是重复手机端记录功能。
- 趋势分析有可观察数据支撑。
- 报告可以用于自我复盘；真实分享、导出或给校医院/药师参考前必须由用户预览确认。

### 场景四：疑似漏服提醒

用户中午需要服药但没有打卡，系统提醒用户确认是否已服药。若用户确认漏服，Luminous 记录漏服事件，并提示查看药品说明书或咨询医生/药师。

成功展示点：

- 系统可以主动识别漏服风险。
- 系统不自动修改药物剂量和服药方案。
- 这个边界能防止产品被质疑为未经监管的医疗决策工具。

## 不做什么

为了让产品在大学竞赛里更聚焦，第一阶段明确不做以下内容：

- 不做运动恢复。
- 不做女性健康或经期管理。
- 不做桌面端。
- 不做智能硬件。
- 不做复杂多端生态。
- 不做心理健康诊断或心理咨询替代。
- 不做医疗诊断。
- 不做处方建议。
- 不做自动改药量、停药、换药或改服药时间。
- 不做没有数据来源支撑的药物相互作用判断。
- 不做未授权的 AI 健康摘要、报告分享或对外导出。

## 未来扩展

未来可以扩展心理健康，但建议从“压力、睡眠、情绪自助和校园资源连接”开始，而不是直接进入心理诊断或治疗。

可扩展方向：

- 情绪和压力记录。
- 睡眠与情绪趋势关联。
- 轻量自助练习。
- 校园心理咨询资源入口。
- 用户可控的隐私隐藏和报告脱敏。

未来也可以扩展更多健康场景，但必须遵守一个原则：

> 新场景必须能接入“记录 -> 总结 -> 主动提醒 -> 报告”的闭环，否则不进入主线。

## 成功标准

第一阶段不要用“功能数量”衡量成功，而要看用户是否真的改变行为。

建议验证指标：

- 用户是否愿意连续 7 天记录至少 2 类健康信息。
- 用户是否认为每日总结比原始记录更有用。
- 用户是否会点击或执行主动提醒。
- 用户新增药物后，是否能理解有来源支撑的安全解释。
- 用户是否愿意在预览和脱敏后，把周报展示给校医院、药师或可信第三方。
- 用户是否认为 Luminous 比临时网页搜索、问同学或手动备忘录更可靠。

如果这些指标不成立，说明问题不在 UI，而在核心价值没有被用户感知。

## 当前产品原则

- 用药安全是可信入口，不是产品天花板。
- AI 应该主动，但不能越权。
- 生活方式建议可以具体，医疗建议必须克制。
- 少做场景，多做闭环。
- 先证明一个校园高频闭环，再扩展更多健康领域。
- 手机端负责行动，Web 端负责报告预览、导出演示和轻量查看。
- 竞赛表达要强调真实用户场景、主动提醒、可解释安全边界和可验证 MVP。
